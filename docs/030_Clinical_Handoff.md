> **状态(2026-07-14)**:方向 + **医生视图版式已定稿**(样张已过评审)。主视图 = **疾病泳道
> 时间轴**(见 §3a)。唯一仍保留的是**「复制病历文本」模版的字段排布** —— 等用户给国内病历
> 模版(《病历书写基本规范》/ 住院病案首页),再把确定性填充对齐;模版是**下游工作流件**,
> 不影响 payload 与时间轴。**已 spin-off 的活 = 术语归一覆盖率**(issue #125):`terminology`
> 已加确定性药名剥壳 + 化验名候选拆分,字典扩容到 ≥98% 是主要工作。抽取一律**确定性、无
> LLM**;LLM 仅可能用于「复制病历文本」的行文润色,且**数字永远由确定性来源填**(见 §5)。

## 0. 本轮定稿(2026-07-14,样张评审通过)

1. **医生视图主轴 = 疾病泳道时间轴(方案 A)**:每个问题一条从起病拉到现在的横条(慢病=横条、急性=圆点),点击任一条**就地展开**该病的化验趋势 + 用药。一眼看「病一个个叠上来」的病程,自解释、免搜索。数据模型直接采用 **FHIR `Condition`**(`onsetDateTime`/`abatementDateTime`/`clinicalStatus`)+ **OMOP `condition_era`**;视觉参照 **LifeLines2 / 甘特泳道**。顶部叠一张**现状总览**(在治疾病 / 当前用药 / 过敏 / 近期变化)。
2. **加密分享 payload = 全量数据 → 结构化 + 我们的分析(时间轴/趋势/异常标记)**,**不是原文全文转储**。这就是医生在查看器里看到的东西。**「模版」≠ payload**:模版是医生看完后点「复制为病历文本」拿走、贴进自己 HIS 的**下游工作流件**。
3. **两种导出故意不对称**:①**手机数据导出(给自己)**= 原始全量、尽量简单,备份/换机用;②**加密分享(给医生)**= 结构化+分析、复杂度花在这。别混。
4. **交付 = E2E 链接,不是 HTML 文件**:HTML 自包含文件**微信发不了/打不开**,链接才是微信的原生分享单位 → 这把 #58(小后端 + 链接 + 真过期)从「1.3 想做」升级为**中国市场刚需**。密钥默认放 URL `#` fragment(浏览器不发给服务器)+ 短有效期 + 可撤销;敏感场景再上口令。服务器**只存密文**(§7 不变量)。
5. **问题→化验/用药分组靠一张带出处的映射表**(见 §3b),**化验按具名指南、用药按 ATC**,不臆造。查看器页脚展示「分组依据」引用清单,作为专业背书。
6. **诚实标注**:时间条起点是「**首次在记录中出现**」,非确诊日(缺早年报告会晚于真实起病),UI 明示。
7. **查看器可用成熟库**:CSP 拦的是**外部域**加载/外发(防泄露),不禁止**同源打包**自己的 JS —— 时间轴要用成熟库就 vendor 进包,不走 CDN(现 sparkline 为手搓 SVG,同理)。



> 战略「第二句话」:第一句是**所有权**——「你的病历只属于你,存在你自己的保险箱里」;
> 第二句是**有用性**——「需要时,它替你把病情讲清楚,并把医生的新判断带回来」。
> 本文是把第二句从口号落成技术方案的设计文档。产品定位与竞品判断见对话记录 2026-07-14。

## 1. 现状 vs 目标

- **现状**:加密分享包 = `{"records":[…逐份文档…]}`,查看器(`web/hosted-viewer` + `packages/share` 内嵌 HTML)把每份文档用内容感知渲染出来。= **给医生看一堆报告**。
- **目标**:在文档列表**之上**加一个**医生首屏摘要**(problem list / 当前用药 / 过敏 / 关键化验+趋势 / 近期就诊),每条**能下钻到源文档**(证据回溯)。= **医生 10 秒看懂病情上下文**。
- **不变量**:证据回溯、E2E(服务器永不见明文/密钥)、患者可撤销 —— 是第二句能不能立住的三条命脉。

## 2. 分三阶段(每阶段独立可交付)

| 阶段 | 内容 | 需要后端? |
|---|---|---|
| **A · 查看器 summary** | 查看器渲染 summary 首屏 + 证据下钻;summary 走**现有自包含加密分享**(发文件+口令) | ❌ 纯客户端 |
| **B · 结构化提取** | app 侧把 OCR 文本抽成结构化临床事实(OMOP-lite 事件),物化出 summary 打进分享包 | ❌(LLM 可选,见 §5) |
| **C · 小服务器** | 发**链接**代替发文件 + **真过期 / 撤销 / 访问日志**(#58) | ✅ 但只存密文 |

**关键:A、B 零后端。** 服务器(C)只买「链接/过期/撤销/日志」四样,是第二阶段,等 summary 跑通再上,顺便去风险。

## 3. 数据契约(THE artifact):doctor-summary JSON

分享包在现有 `records` 之外新增 `summary` 字段(缺省则查看器回退到纯文档列表,老分享不受影响)。`evidence` 是 `records` 数组的下标,点击跳到源文档 → 证据回溯白拿。

```jsonc
{
  "records": [ /* 现有:逐份文档(doc_type, ocr text, dicom…) */ ],
  "summary": {
    "generated_at": "2026-07-14T…",
    "patient": { "name": "张建国", "gender": "男", "age": "58" },
    "problems": [
      { "term": "2型糖尿病", "onset": "2021-05", "status": "active",
        "acute": false, "evidence": [3, 7] },
      { "term": "急性肺炎", "onset": "2022-11", "end": "2022-12",
        "status": "resolved", "acute": true, "evidence": [4] }
    ],
    "medications": [
      { "name": "二甲双胍", "dose": "0.5g bid", "status": "active",
        "since": "2024-01", "note": null, "evidence": [7] },
      { "name": "阿司匹林", "dose": "100mg qd", "status": "stopped",
        "since": "2023-06", "note": "2024-03 停用", "evidence": [2, 5] }
    ],
    "allergies": [
      { "substance": "青霉素", "reaction": "皮疹", "evidence": [3] }
    ],
    "key_labs": [
      { "analyte": "HbA1c", "unit": "%",
        "latest": { "value": "7.2", "date": "2026-06", "flag": "high" },
        "trend": [ {"date":"2025-12","value":"7.8"}, {"date":"2026-06","value":"7.2"} ],
        "evidence": [9] }
    ],
    "recent_encounters": [
      { "date": "2026-06-15", "provider": "协和内分泌", "kind": "门诊", "evidence": [9] }
    ],
    "notable_changes": [ "近半年 HbA1c 7.8→7.2", "阿司匹林 2024-03 停用" ]
  }
}
```

- **status = active|stopped|unknown**:当前用药靠对 drug 事件按日期 fold(见 §4)得出;冲突(停了又用)落在 `note`。
- **problem.onset / status / acute**:驱动 §3a 泳道时间轴。`onset` = **首次在记录中出现**(非确诊日,UI 明示);`acute:true` → 画成圆点而非横条;`status: resolved` → 灰化/可折叠。
- **不做**:标准词表编码(RxNorm/LOINC 概念 id)。医生看中文药名/指标名即可;编码是去重聚合 + Prometheno 互通用的,推后(§6)。

## 3a. 医生视图渲染契约:疾病泳道时间轴

查看器把上面的 `summary` 渲染成:**顶部现状总览卡**(problems / active 用药 / allergies / notable_changes)+ **疾病泳道时间轴**主体 +(病多时)**筛选 chip**(全部 / 只看在治 / 只看异常)。

- **泳道**:每个 `problem` 一行,横条从 `onset` 到 `end`(缺省=现在);`acute` → 圆点。X 轴 = 时间,等距年份网格线。**自上而下按 onset 由早到晚**,看病程叠加。
- **下钻**:点击一条 → 就地展开该病的**相关化验(趋势 sparkline)+ 相关用药**。「相关」由 §3b 映射表把 `key_labs` / `medications` 归到各 `problem` 名下;**归不上的进「其他」,不丢**。
- **图示 legend**:横条/圆点/正常点/异常点含义显式标注。
- **复制病历文本**(模版,下游件):总览卡上一枚按钮,把结构化事实确定性填成一段可粘进 HIS 的文本;字段排布等用户给国内病历模版后定。
- **抗量**:泳道每病一行,比展开卡列表更省纵向空间;20+ 病靠「在治优先 + 筛选 + 已愈折叠」压住,各病趋势图独立不互挤。

## 3b. 问题→化验/用药 映射表(分组引擎)

把 `key_labs` / `medications` 归到各 `problem` 名下的**确定性数据表**,原则:**不臆造,每条带出处**。

- **药 → 疾病**:以 **ATC 治疗学分类**前缀为准(ATC 前缀即引用)。例:`A10*`→糖尿病、`C10*`→高脂、`C07-C09`→高血压。药已带 ATC 码(复用 `packages/terminology` 字典)。
- **化验 → 疾病**:LOINC 只说「是什么化验」不说「属哪个病」→ 以**具名临床指南(含年份)或 WHO/OMOP phenotype** 为据。例:HbA1c/空腹血糖→《中国糖尿病防治指南(2024版)》。
- **展示**:查看器页脚「分组依据」列出所有引用标准(ATC + 各指南),作为专业背书 + 「我们不猜」的证明。
- **落地**:草表 `problem_map.draft.json` + 方法/引用 `problem_map.methodology.md` **已产出并审过**(现在 scratchpad,构建时落入 repo 定版),覆盖 10 个常见慢病:糖尿病/高血压/高脂/冠心病/CKD/甲减/甲亢/贫血/痛风/脂肪肝;58 化验行 + 37 药物行。冠心病/贫血/脂肪肝的薄弱处已在表内如实标注。

## 3c. 复制病历文本 · 院外就诊经历模板(模版)

医生看完 summary 后一键「复制为病历文本」拿走、贴进自己 HIS 的**下游工作流件**(≠ payload)。用户 2026-07-15 提供的真实模板:

> 患者曾于[YYYY年MM月DD日]于[外院名称][住院/门诊]治疗。[影像学1名称]提示[具体描述];[影像学2名称]提示[具体描述];[病理/检验名称]提示[具体描述]。考虑诊断为[疾病名称],予[手术名称/药物名称及剂量]治疗。术后病理提示[具体病理结果]。
> [YYYY年MM月DD]于[外院名称]复查。[影像学1名称]提示[具体描述];[检验名称]提示[具体描述]。目前系[术后/化疗后/靶向治疗后]状态,现规律服用[现用药名及剂量]治疗,效果[较好/一般/较差]。患者为[进一步诊治/复查/咨询治疗方案]来我院就诊。

**生成三原则(用户定)**:
1. **化验只放主要阳性结果 + 变化趋势**,不堆全部(如血常规一般只 RBC/PLT/WBC/Hb)。实现:阳性(超参考区间)优先排序,值以 `首次→最近` 带 ↑/↓ 呈现。
2. **影像 / 病理是重点**,叙事里前置。
3. **在合理范围内信息宁多勿少** —— 对医生删繁比补漏容易。缺项自动略过,不硬凑。

**实现 = 确定性模板填充,无需模型**:这是固定模板,槽位全是结构化数据(日期/机构/影像描述/诊断/治疗/用药+剂量/效果/来诊原因)。数字永远取自结构化来源,**模型永不改数**;未来若要脱离固定模板做自由文本综述,才引入端上小模型润色(仍不碰数字)。见查看器 `buildEMRFromSummary`。

**为此 summary 增补的可选字段**:`care_facility`、`imaging:[{date,name,finding}]`、`pathology:[{date,name,finding}]`、`status_note`(目前系…状态)、`effect`(较好/一般/较差)、`visit_reason`。影像/病理的「具体描述」= 报告结论段,由阶段 B 提取。

## 4. OMOP-lite 结构层(阶段 B 的地基)

采用 OMOP 的**领域形状**、跳过标准词表。每类事实建成**事件**进现有 append-only 日志(白拿证据回溯 + 跨设备合并 + 冲突物化):

| OMOP 域 | MedMe 事件载荷 |
|---|---|
| drug_exposure | {药名, 剂量, 频次, 动作(start/continue/stop), 日期, source_file_hash} |
| measurement | {指标, 值, 单位, 参考范围, 高低标记, 日期, source} |
| condition | {诊断词, 日期, source} |
| observation | {物质, 反应?, source}(过敏等) |
| procedure | {名称, 日期, source} |
| visit | 复用现有 encounter |

**当前用药 = fold**:阿司匹林 start(d1)→ stop(d2)→ restart(d3) ⇒ `active(自 d3),note: d2 停`。**只有事件溯源能干净做,纯 AI 摘要做不到 —— 这是独有能力。**

## 5. 抽取:parser 先,LLM 后(要拍板的分叉)

- **化验**:本来就有表格解析 → **持久成结构事实**(不只显示)。走 parser。
- **诊断/过敏/病程**(叙述性)→ LLM 结构化抽取更稳。隐私分叉:端上小模型(纯本地/弱) vs 服务器 LLM(强/OCR 文本出设备)。
- **务实**:混合;且 **LLM 抽取只在「生成医生分享」那一刻跑**(用户已决定分享,是可接受一次 AI 调用的时机)。平时纯本地。
- **MVP 不引入 LLM**:先用 parser/启发式从现有 `records` 抽 用药+化验+过敏,证明管线。

## 6. 与 Prometheno 的桥

结构层做成 OMOP 形状(哪怕不带词表)→ 患者数据将来流进 Prometheno 的 OMOP testbed **天然兼容**。MedMe = 患者侧的 OMOP-lite 喂数端。第三句(整个就医网络的受控协作)靠第二句攒下的医生网络效应,现在不做。

## 7. 服务器边界(阶段 C,#58)

只干四件事,**E2E 不变量:只存密文,永不见密钥/明文**:
- 存加密分享包(密文 blob)→ **发链接代替发文件**(HTML 文件微信发不了/打不开,链接才是微信原生分享单位 → #58 是中国市场刚需,非可选)
- 密钥默认放 URL `#` fragment(浏览器不发给服务器,服务器仍只见密文)+ 短有效期;敏感场景再上口令(老人有摩擦,默认零摩擦)
- 真过期(到期删密文)· 撤销(患者一键删)· 访问日志(谁何时打开)
- 合规面最小:服务器上只有密文,PIPL/医疗数据暴露面 = 0 明文

后续:医生「建议更正」= 一个 proposed 事件回传,患者确认后写入 → 从「发病历」进化到「共同维护病历」(第二句最重的半句)。

## 8. 最小切片(现在就做)

**查看器先行,零后端、零 app 改动**:
1. 定死 §3 的 `summary` schema(本文即契约);版式按 §3a(泳道时间轴 + 总览 + 筛选)。样张已过评审(scratchpad `trends-mock.html`)。
2. 查看器加 `renderSummary(payload.summary)`:顶部现状总览 + 疾病泳道时间轴,点击泳道下钻该病化验趋势+用药,`evidence` → 滚动到对应 `records[i]`。`summary` 缺省则完全回退现状(老分享不受影响)。
3. 用 sample payload 验 UX(样张已验)。
4. 跑通后:app 侧做阶段 B 的 parser 抽取 + §3b 映射分组 → 产出 `summary` 打进分享包;再上阶段 C 服务器(链接/过期/撤销)。

顺序:**查看器(A)→ app 提取(B)→ 服务器(C)**。先证明医生觉得有用,再补撤销/链接。
