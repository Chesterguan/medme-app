## 送达方案:实测数字定的方向

争论点是「医生怎么打开分享」。用真实语料量完,方向自己浮出来了:

| 通道 | 容量上限 | 要服务器 | 医生能看到 |
|---|---|---|---|
| **二维码**(面对面扫) | ~2.9 KB | 只要静态托管 | 有界泳道 + 趋势 + 在用药 |
| 链接 fragment(微信发) | ~7–32 KB,微信识别存疑 | 只要静态托管 | 全部结构化,无原件 |
| **文件**(U 盘 / 微信传文件) | 300 MB | 不要 | 全部 + 原件 + DICOM |
| 小服务器(只存密文) | 无实际上限 | 要 | 全部 + 真过期/撤销 |
| **患者手机当场翻** | 无 | 不要 | 全部(在患者设备上) |

**关键实测**:未裁剪的 summary 体积随病历份数增长,**80–100 份就撑破二维码** —— 慢病随访患者一定撞上,而那正是最需要 MedMe 的人群。所以载荷**按构造裁剪**(`QrLimits`),裁剪后体积与病历总量脱钩:5 个疾病 853 字符 → 500 个疾病 865 字符(合成样本,`qr::tests::print_sizes` 可复现)。

真实示例数据集实测:6 个疾病 · 密文 1248 字节 · URL 1734 字符。

**容量上限是 2331 而非 2953** —— 2953 是纠错等级 L,出码端实际用等级 M(隔着桌子扫需要更高纠错)。
一度两边不一致(守卫按 L、渲染按 M,宽了 27%),已在 #169 修正。**照此重算:真实数据已用掉
74% 容量,只剩约 34% 的增长空间** —— 调 `QrLimits` 时这就是预算。

**两处被实测推翻的直觉**:
- 「微信发文件就行」—— 用户实测:能收到,**iOS 上没有一键打开的办法**。
- 「用小服务器才装得下大数据」—— 不成立,文件通道零服务器也能带原件。服务器真正买到的是**真过期与真撤销**(现在的「有效期」只是提醒,拦不住任何人)。

分阶段计划(含未决议项)在 Notion「医生分享送达方案」。技术设计待会议定论后落 ADR。

## 阶段一已落地(#164 + #166)

保险箱 → 装配 summary → 按 `QrLimits` 裁剪 → gzip → AES-256-GCM → URL fragment → 手机出码 → 医生扫 → 查看器解密渲染。

- fragment 带 `q1.` 版本前缀,查看器据此区分「二维码载荷」与「口令」。
- AAD 用 `medme-qr-v1`,与整份分享的 `medme-share-v1` **刻意不同** —— 两种载荷不会被互相当成对方解开(有测试)。
- gzip 解压用浏览器原生 `DecompressionStream`,不引前端库。
- **密钥同在 fragment,不再要口令**:面对面出示码本身即「在场证明」。代价写进了界面:「这张码就是钥匙」。
- 顺带落地的生产能力:**口令可随 URL fragment 送达**并在到达后清除 hash(解析期与 load 时清都会被浏览器随后的片段导航覆盖,需补一次延迟清除 —— 实测踩到)。

`QrLimits` 默认值(6 病 / 每病 2 化验 / 每化验 4 点 / 3 在用药 / 3 条近期变化)是**工程占位**,等真机 UI/UX 反馈后改常量。

## 模糊测试:第一次跑就挖到真问题(#165)

三个 target(`dicom_parse` / `parser_text` / `summary_assemble`)+ CI 每次 PR 跑 120 秒。

**发现**:DICOM 元数据解析存在**内存耗尽(DoS)** —— 上游 `dicom-rs` 按数据元素里声明的长度直接分配,发生在**解析期**,早于既有的 `check_decode_bounds`。数 KB 的畸形文件即可诱导数 GB 分配。攻击面很实:医院光盘、他人转发的文件。

**三道防线**(见 [#165](https://github.com/Chesterguan/medme/pull/165) 与后续提交):
1. `dicom::prescan_lengths` —— **不分配、只前进**的浅扫,声明长度超过文件剩余即拒绝。主进程可安全调用。
2. 元数据解析移进**隔离子进程**(桌面端;`pipeline` 加注入点 `ingest_with_dicom_parser`,照 share 的 `DicomPngRenderer` 先例)。
3. CI 短时模糊测试常态化。

**尚未彻底关闭**:浅扫遇未定长序列等结构会交回上游,根因在上游按声明长度分配。需上游修复或更强隔离。

**上游提报:已查清,暂缓。** 调查结论是**缺失的加固而非设计取舍** —— `try_reserve` 在 `dicom-object/src/meta.rs`
有 7 处、在整个 `dicom-parser` 有 0 处;维护者自己的 PR #259 修的就是同类问题(「prevents panicking from OOM
caused by malformed DICOM data」);上游 PR #800 正在给 parser 补 fuzz target,说明这条路此前没被扫过。
报告草稿在 `docs/upstream-dicom-rs-report.md`(**已 gitignore**,含未公开细节)。按他们 `CONTRIBUTING.md:32`
的要求需人工核验后由本人提交,**当前决定先不提**。

影响面:移动端零改动(文件选择器不接受 `.dcm`)、查看器不受影响(那是另一套 JS 解析器)、功能零变化。

## 追记(2026-07-20):扫描版 PDF 的真机反馈与一次自伤

阶段一二维码在 iOS 真机验证通过(v20)。同批验证的**扫描版 PDF 却不合格**,而修的过程
本身值得留痕。

**v20 真机反馈**:识别「内容不对,排版问题更大」。

**v21(#173)查出两处,但引入了更严重的回归**:

1. 渲染解析度。此前写死 `page.width * 2`,我按「pdfx 的 width 单位是 point」换算成
   144 DPI,认为低于中文 OCR 通行下限 300 DPI,于是改成 4.17×。
2. 页间用 `writeln` 单换行拼接,把上一页末块与下一页首块粘成同一块 —— OCR 本身是按块
   输出、块间 `\n\n` 的(Layer-0),这里破坏了它。第 2 条是真修复。

**但第 1 条是错的**:pdfx 的文档写明 `page.width` 是**像素**不是 point
(`interfaces/page.dart:46`「Page source width in pixels」)。我没查就当成 point,
放大倍数远超预期,`render()` 在真机上失败返回 null,而 `if (img == null) continue;`
**静默跳过** —— v21 从「识别不准」变成「完全识别不出」。

**v22(#175)的改法换了思路**:不再赌单一常量,**渲染倍数逐级降档 3× → 2× → 1.5×,
第一个渲得出来的就用**,每档失败都打日志。设备内存各异,写死一个数就是在赌,赌输了
整页无文本还静默。同时补上**白底** —— `render()` 的 PNG 默认透明背景,黑字压透明底被
OCR 加载时可能合成成黑底,这个 v20 就存在。

**教训两条**:
- 又一次「拿没验证的当事实」——`page.width` 的单位查一眼就有,我却先算了一整张 DPI
  换算表。算术没错,前提是错的,而正确的算术让错误的前提显得可信。
- **静默失败是放大器**。`if (img == null) continue;` 让一个渲染错误变成了「整篇零文本
  且无任何提示」。改动涉及真机行为时,失败路径必须有日志。
